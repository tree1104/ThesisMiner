## v1.0 - v3.0：初始架构与基础功能

### v1.0 - Genesis（2024-03-15）

#### 概述

ThesisMiner v1.0 是项目的初始版本，提供了最基础的论题生成功能。这是一个命令行工具，用户通过输入学科方向和研究兴趣，获得 AI 生成的论题建议。

#### 核心特性

1. **基础论题生成**
   - 单一模型支持（OpenAI GPT-3.5）
   - 基于模板的 Prompt
   - 一次生成 5 个论题
   - 简单的文本输出

2. **命令行界面**
   - 交互式输入
   - JSON 格式输出
   - 基础错误处理

#### 技术栈

| 组件 | 技术 |
|------|------|
| 语言 | Python 3.9 |
| AI 模型 | OpenAI GPT-3.5-turbo |
| HTTP 客户端 | requests |
| 配置 | python-dotenv |
| 测试 | unittest |

#### 代码结构

```
thesisminer/
├── main.py              # 主入口
├── generator.py         # 论题生成器
├── prompts.py           # Prompt 模板
├── config.py            # 配置
├── .env.example         # 环境变量示例
└── README.md            # 说明文档
```

#### 示例用法

```bash
$ python main.py
请输入学科方向：计算机科学
请输入研究兴趣：机器学习，深度学习

正在生成论题...

1. 基于深度学习的图像识别方法研究
2. 机器学习在自然语言处理中的应用
3. 深度学习模型的优化算法研究
4. 基于机器学习的数据挖掘方法
5. 深度学习在语音识别中的应用
```

#### 局限性

- 仅支持单一模型
- 无 Web 界面
- 无数据库持久化
- 无缓存机制
- Prompt 固定，无法定制
- 无约束校验
- 无成本统计

### v2.0 - Foundation（2024-05-20）

#### 概述

v2.0 引入了 Web 界面和 REST API，使 ThesisMiner 从命令行工具转变为 Web 服务。

#### 核心特性

1. **Web 界面**
   - 基于 Flask 的简单前端
   - 表单输入
   - 结果展示页面
   - 基础样式

2. **REST API**
   - `POST /api/generate`：生成论题
   - `GET /api/health`：健康检查

3. **配置化**
   - 支持通过环境变量配置
   - 模型参数可调

### v5.0 - CacheEngine（2025-02-18）

#### 概述

v5.0 引入了缓存架构，显著提升了响应速度并降低了 API 成本。

#### 核心特性

1. **多级缓存**
   - 内存缓存（LRU）
   - 磁盘缓存（SQLite）
   - 缓存键基于请求参数哈希

2. **性能优化**
   - 异步数据库操作
   - 连接池
   - 查询优化
   - 索引优化

3. **成本统计**
   - Token 使用量统计
   - 成本计算
   - 日/周/月报表

#### 缓存架构

```
请求 → 缓存查询 → 命中？ → 是 → 返回缓存
                ↓否
          调用 API → 存入缓存 → 返回结果
```

#### 缓存效果

| 场景 | 无缓存延迟 | 有缓存延迟 | 提升 |
|------|-----------|-----------|------|
| 相同请求 | 2.5s | 0.05s | 50x |
| 相似请求 | 2.5s | 1.2s | 2x |
| 不同请求 | 2.5s | 2.5s | - |

#### 成本统计示例

```python
# v5.0 成本统计
class CostTracker:
    def __init__(self):
        self.total_tokens = 0
        self.total_cost = 0.0

    def record(self, input_tokens: int, output_tokens: int, model: str):
        cost = self._calculate_cost(input_tokens, output_tokens, model)
        self.total_tokens += input_tokens + output_tokens
        self.total_cost += cost

    def report(self) -> dict:
        return {
            "total_tokens": self.total_tokens,
            "total_cost": self.total_cost,
            "avg_cost_per_request": self.total_cost / self.request_count
        }
```

#### 检索集成

```python
# v6.0 检索集成
class SearchIntegrator:
    def __init__(self):
        self.sources = [
            SemanticScholarSource(),
            ArxivSource()
        ]

    async def search(self, query: str) -> list:
        results = []
        for source in self.sources:
            results.extend(await source.search(query))
        return self._deduplicate(results)

    def _deduplicate(self, results: list) -> list:
        # 基于标题相似度去重
        seen = set()
        unique = []
        for r in results:
            key = self._normalize_title(r["title"])
            if key not in seen:
                seen.add(key)
                unique.append(r)
        return unique
```

## v8.0：多 Agent 架构与五阶段流程

### v8.0 - Horizon（2026-06-15）

#### 概述

v8.0 是 ThesisMiner 迄今为止最大的一次版本更新，引入了全新的多 Agent 架构、五阶段闭环导航流程、多对话管理、D3.js 谱系图谱等核心特性。

#### 核心特性

1. **多 Agent 架构**
   - 4 个专业化 Agent（Reasoner/Mentor/Searcher/Critic）
   - Agent 编排状态机
   - Hook 机制
   - 自定义 Agent 支持

2. **五阶段闭环导航流程**
   - 信息确权 → 创意生成 → 校验回退 → 多粒度生成 → 深度辅助
   - 四维创意引擎
   - 校验失败自动回退

3. **多对话管理**
   - 同一会话多个并行对话
   - 上下文隔离
   - DST 压缩

4. **D3.js 谱系图谱**
   - 交互式可视化
   - 多种布局算法
   - 节点/边类型丰富
   - 导出功能

5. **DeepSeek 缓存优化**
   - SHA-256 前缀哈希
   - 三段式 Prompt
   - 68.5% 缓存命中率

6. **透明账本**
   - 每次 API 调用记录
   - 多维度统计
   - 成本告警

7. **联网搜索展示**
   - 实时检索
   - 过程可视化
   - 来源标注

8. **安全加固**
   - API Key 加密
   - CORS 配置
   - 速率限制
   - 审计日志

## 附录 G：各版本技术决策记录

### G.1 v1.0：为何选择 Python

**决策**：使用 Python 作为开发语言

**背景**：项目初期需要快速验证概念，AI API 调用以 Python 生态最为成熟。

**替代方案**：
- Node.js：异步 IO 好，但 AI 库不如 Python 丰富
- Go：性能好，但开发速度慢，AI 生态弱
- Java：企业级，但过于笨重

**结果**：Python 的 AI 生态（OpenAI SDK、LangChain 等）和快速开发能力使其成为最佳选择。

### G.2 v2.0：为何选择 Flask

**决策**：使用 Flask 作为 Web 框架

**背景**：v2.0 需要快速添加 Web 界面，Flask 是最简单的选择。

**替代方案**：
- Django：功能全但过于重量级
- FastAPI：当时还不成熟
- Tornado：异步但社区小

**结果**：Flask 满足了初期需求，但在 v4.0 被替换为 FastAPI。

### G.3 v4.0：为何迁移到 FastAPI

**决策**：从 Flask 迁移到 FastAPI

**背景**：v3.0 后，性能瓶颈显现，需要异步支持；同时需要自动 API 文档。

**迁移原因**：
1. **异步支持**：Flask 的同步模型限制了并发能力
2. **自动文档**：FastAPI 自动生成 OpenAPI 文档
3. **类型安全**：Pydantic 集成提供数据验证
4. **性能**：FastAPI 基于 Starlette，性能优于 Flask
5. **现代设计**：依赖注入、中间件等现代特性

**迁移成本**：约 2 周开发时间，重写了所有 API 端点。

**结果**：迁移后并发能力提升 5 倍，API 文档自动生成，开发效率提升。

### G.4 v5.0：缓存策略选择

**决策**：采用多级缓存（内存 + 磁盘）

**背景**：API 调用成本高，响应慢，需要缓存。

**替代方案**：
- Redis：功能强但需额外部署
- 纯内存：重启丢失
- 纯磁盘：速度慢

**选择理由**：
1. 内存缓存提供快速访问
2. 磁盘缓存（SQLite）提供持久化
3. 无需额外依赖
4. 适合中小规模部署

**结果**：缓存命中率约 30%，成本降低 25%。

### G.5 v6.0：约束系统设计

**决策**：硬约束与软约束分离

**背景**：论题质量不稳定，需要约束机制。

**设计原则**：
1. **硬约束**：必须满足的条件，失败即拒绝（fail-fast）
2. **软约束**：评分制，影响排序但不拒绝
3. **可扩展**：支持自定义约束
4. **可组合**：支持 AND/OR/PRIORITY 组合

**结果**：论题质量显著提升，不合格论题减少 80%。

### G.6 v7.0：多模型架构

**决策**：引入多模型注册与步骤路由

**背景**：单一模型无法满足所有场景，不同模型有不同优势。

**设计要点**：
1. **统一接口**：所有模型实现相同接口
2. **步骤路由**：按任务阶段选择模型
3. **Fallback**：主模型失败时自动切换
4. **A/B 测试**：支持模型对比

**结果**：成本降低 40%，质量提升 15%。

### G.7 v8.0：多 Agent 架构

**决策**：引入多 Agent 架构与五阶段流程

**背景**：单一引擎难以处理复杂的论题生成流程，需要专业化分工。

**设计要点**：
1. **专业化 Agent**：每个 Agent 专注一个领域
2. **状态机编排**：OrchestrationStateMachine 管理流程
3. **Hook 机制**：支持在关键节点插入自定义逻辑
4. **上下文隔离**：每个 Agent 独立上下文
5. **可扩展**：支持自定义 Agent

**挑战**：
- Agent 间通信复杂度增加
- 状态管理更复杂
- 调试难度增加

**结果**：论题质量提升 35%，成本降低 68%（缓存优化）。

### I.1 v1.0 回顾

**问：v1.0 的开发过程中最大的挑战是什么？**

答：最大的挑战是在有限的时间内验证概念。我们只有 2 周时间，需要证明 AI 生成论题的可行性。最终我们选择最简单的方案：单一模型、命令行界面、无数据库。虽然简陋，但验证了核心想法。

### I.2 v4.0 重构

**问：为什么选择在 v4.0 进行大规模重构？**

答：v3.0 后我们意识到 Flask 的同步模型会成为瓶颈。同时，前端用 HTML + Bootstrap 维护困难。我们决定"长痛不如短痛"，一次性解决后端和前端的问题。虽然重构花了 1 个月，但为后续版本的快速迭代打下了基础。

### I.3 v8.0 多 Agent 架构

**问：v8.0 的多 Agent 架构是如何设计的？**

答：我们参考了 AutoGPT 和 LangChain 的 Agent 设计，但做了重要改进：
1. 每个 Agent 有明确的职责边界
2. 使用状态机管理流程，而不是自由编排
3. Hook 机制允许在不修改核心代码的情况下插入逻辑
4. 上下文隔离确保 Agent 间不互相干扰

最大的挑战是平衡灵活性与可控性。我们最终选择了"状态机 + Hook"的方案，既保证了流程的确定性，又保留了扩展空间。
