# Claude Code 架构设计全面解析与演进（2026 深度修订版）

> 本文档基于 Claude Code 源码逆向工程、底层通信协议解析、官方工程实践及最新社区生态发展，对系统架构进行深度修正与万字号补充。从系统提示词的概率空间驾驭、上下文与缓存的高可用状态机、深度防御安全模型、复杂智能体拓扑到跨会话记忆网络，梳理出一套面向生产级 AI Agent 的可复用工程架构模式。

## 核心架构哲学：Kernel Harness 架构与极简闭环

Claude Code 的核心运行时是一个“极薄的 AI 内核包在极厚的确定性外壳里”的 **Kernel Harness** 架构。整个系统的运行核心是一个简洁到令人发指的 `while` 循环：

```typescript
// 极简化的核心运行时伪代码
while (true) {
  1. 构建上下文 (System Prompt + Tools + History + Memory)
  2. 调用 LLM API
  3. 解析 LLM 响应 (区分文本与工具调用)
  4. 权限校验与沙箱执行
  5. 将工具结果追加至上下文
  6. 判断是否结束
}
```

模型负责“思考”与决策（占代码量 1.6%），而 Harness 负责“兜底”（占 98.4%）。没有复杂的多步编排图或硬编码状态机，所有的工程基础设施（权限管理、上下文压缩、会话存储、工具调度）都是为了确保这个循环在开放性与安全性之间达到极致平衡。

## 一、系统提示词与工程化引导架构

### 1.1 分段式组合架构与缓存摩擦设计
系统提示词并非一段长文本，而是被拆分为独立的、可记忆化的段落，通过注册表管理生命周期。每次构建请求时，系统按序拼接这些段落。

```typescript
type SystemPromptSection = {
  name: string
  compute: ComputeFn  // () => string | null | Promise
  cacheBreak: boolean // false = 可记忆化, true = 每轮重算
}
```
系统提供两个工厂函数以区分内容稳定性：
- `systemPromptSection(name, compute)`: 记忆化，结果缓存到全局状态，直到显式清除。
- `DANGEROUS_uncachedSystemPromptSection(name, compute, reason)`: 每轮重算，破坏缓存。

**设计摩擦**：`DANGEROUS_` 前缀和强制的 `reason` 参数是有意为之的 API 设计。在代码审查时，开发者必须看到这个刺眼的前缀并撰写理由，迫使开发者在破坏昂贵的 API 缓存前权衡成本。

### 1.2 六种行为引导模式：概率空间驯化
系统提示词通过精心措辞的行为指令在模型的概率空间中设定边界，防止模型产生常见的“AI 脑残”行为：
1. **极简主义指令**：明确禁止过度工程，给出具体数量门槛（如 *"Three similar lines of code is better than a premature abstraction"*）。利用具体数字锚定，避免模型对“简单”的主观理解产生偏差。
2. **渐进式升级**：强制从简单方案开始，逐步增加复杂度，防止模型一开始就设计庞大的重构计划。
3. **可逆性意识**：强调操作的可逆性，降低破坏性风险。引导模型优先选择可撤销的操作（如创建新文件而非覆盖旧文件）。
4. **工具偏好引导**：将流量导向专用工具。明确映射表：文件搜索→`Glob`，内容搜索→`Grep`，避免模型用 `Bash` 的 `find` 和 `grep` 凑合，从而绕过沙箱限制和遥测追踪。
5. **Agent 委托指引**：明确何时派生子 Agent。指导模型在遇到“需要大量上下文的独立子任务”时分裂子 Agent，保护主上下文窗口。
6. **数值锚定**：用具体数字锚定模型的判断基准（如“不超过 5 个文件”、“至少 3 次测试”）。

### 1.3 工具提示词作为微型驾驭器
每个工具的 `description` 字段被扩展为一套完整的行为约束协议，包含：功能描述、正面引导、负面禁令、条件分支、格式模板。

以最复杂的 `BashTool` 为例，其提示词构成一套严密的微操协议：
- **工具偏好矩阵**：明确告知模型优先使用专用工具而非通用 Bash 命令（如用 Read 代替 `cat`，用 Edit 代替 `sed`）。
- **命令拼接规则**：`&&`（有依赖，前序成功才执行）→ `;`（允许失败，顺序执行）→ **禁止换行符**（防止注入和多步执行歧义）。
- **Git 安全协议（六层防线）**：
  1. 绝对禁止 `push --force` 到受保护分支。
  2. 绝对禁止 `reset --hard` 到远程已有提交。
  3. 绝对禁止 `checkout .` 批量丢弃工作区改动。
  4. 提交前必须检查 `git status` 防止提交敏感文件。
  5. 合并冲突时禁止盲目 `--theirs` 或 `--ours`。
  6. 推荐使用 `--no-verify` 必须在 Hook 中显式声明理由。

### 1.4 API 通信韧性层与控制平面
通信层不是简单的 SDK 封装，而是 Agent 的控制平面，负责处理网络抖动、限流和模型过载。

**重试策略核心常量矩阵：**
| 常量 | 值 | 用途 |
| :--- | :--- | :--- |
| `DEFAULT_MAX_RETRIES` | 10 | 默认重试预算 |
| `MAX_529_RETRIES` | 3 | 连续 529 过载后触发模型降级（如从 Opus 降到 Sonnet） |
| `BASE_DELAY_MS` | 500ms | 指数退避基数（计算公式：`500 * 2^(attempt-1)`） |
| `PERSISTENT_MAX_BACKOFF_MS` | 5 分钟 | 持久模式最大退避上限 |
| `HEARTBEAT_INTERVAL_MS` | 30 秒 | 防止容器空闲回收（发送空请求保活） |

**可观测性与错误分类**：每次 API 调用发出三个遥测事件（请求发出、成功、失败），配合 25 种细粒度错误分类（如 `RATE_LIMITED`, `CONTEXT_TOO_LONG`, `INVALID_API_KEY`, `OVERLOADED` 等）。当连续遭遇 529 过载时，系统会判定当前模型不可用，自动触发模型降级逻辑，保证主循环不被阻断。

### 1.5 模型特定调优与发布演进
- **`@[MODEL LAUNCH]` 注解**：作为分布式检查清单，全局搜索即可定位所有需适配新模型的代码点，确保新模型上线时所有硬编码的行为约束得到同步更新。
- **行为缓解机制**：针对特定模型（如 Capybara v8）的过度注释、幻觉加剧等四个已知问题，在提示词层施加针对性的负面禁令缓解措施。
- **GrowthBook A/B 测试与 Undercover 模式**：用于内部用户无感验证新模型表现，通过特征开关动态路由请求到新模型，对比用户满意度指标。

## 二、上下文与缓存高可用状态机

### 2.1 提示缓存的“行李箱打包”布局
LLM API 调用的成本和延迟极大约束在于前缀匹配。为追求极致的速度与成本（缓存命中可降低 90% 成本与延迟），架构围绕一段不可变的文本前缀设计请求结构。

**三级缓存范围与 TTL 设计：**
```typescript
{ type: 'ephemeral' }  // 标准缓存（5分钟）
{ type: 'ephemeral', scope: 'global' | 'org', ttl: '5m' | '1h' }  // 扩展缓存
```

**行李箱分层模型**：
1. **最外层（全局静态）**：系统提示词约 11K tokens + 40+ 工具 Schema 约 20K tokens = 每次调用 30K+ tokens 固定开销。跨所有用户复用，缓存命中率极高。
2. **次外层（项目级）**：`CLAUDE.md` 项目规则，同项目复用。
3. **中间层（会话级）**：当前会话上下文与记忆。
4. **最内层（动态）**：不断增长的对话消息历史。

API 请求序列化顺序为 `[系统提示词] → [工具定义] → [消息历史]`，任何前缀变化都会导致后续所有缓存中断。

### 2.2 两阶段缓存中断检测机制
为了诊断“为什么缓存没命中”，架构设计了严密的检测机制：
- **阶段 1（请求前）**：捕获当前状态快照，与前次状态进行深度 diff，记录变化清单（如“GrowthBook 特征开关翻转”、“日期跨天”）。
- **阶段 2（响应后）**：检查 API 返回的 `cache_read_input_tokens` 字段，确认是否真正中断。如果中断，用阶段 1 的变化清单生成可读的诊断日志。

**关键工程细节**：
- 排除 `defer_loading` 工具（API 会自动剥离这些不活跃工具的 Schema，不影响缓存键）。
- 传入**锁存后的值**而非实时状态。例如，如果在请求构建过程中用户切换了 Fast Mode，系统不会立即应用，而是锁存当前状态，保证本次请求的缓存键稳定，下一次请求再生效。

### 2.3 七个命名缓存优化模式
为了最大化缓存命中，系统采用七种策略将动态内容静态化：
1. **日期记忆化**：日期跨天变化导致缓存中断 → 使用 `memoize(getLocalISODate)` 保证同一天内日期常量不变。
2. **月度粒度**：日期每日变化 → 使用 "Month YYYY" 而非完整日期，将缓存有效期从 1 天延长至 30 天。
3. **Agent 列表附件化**：Agent 列表动态变化 → 从工具描述（系统提示词层）移至消息附件（历史层），避免系统提示词变动。
4. **技能列表预算**：技能数量增长 → 限制技能描述总体积不超过 context window 的 1%。
5. **`$TMPDIR` 占位符**：用户 UID 嵌入路径导致跨用户缓存中断 → 替换为 `$TMPDIR` 环境变量占位符。
6. **条件段落省略**：功能开关改变提示词 → 条件性省略整个段落而非添加新内容，保持前缀不变。
7. **工具 Schema 缓存**：GrowthBook 翻转/动态内容 → 使用会话级 `Map` 缓存工具定义，避免每轮重算。

### 2.4 自动压缩与快照-清空-恢复模式
当 Token 数达到 `autoCompactThreshold` 时触发自动压缩。

**阈值计算公式：**
```text
effectiveContextWindow = contextWindow - reservedTokensForSummary
autoCompactThreshold = effectiveContextWindow - AUTOCOMPACT_BUFFER_TOKENS
reservedTokensForSummary = min(maxOutputTokens, 20_000) // p99.99 压缩摘要上限
AUTOCOMPACT_BUFFER_TOKENS = 13_000 // 安全缓冲区
```
设计哲学为：*"The best compression is the one the user never notices."* 用户不应感觉到上下文被截断。

压缩前执行**快照-清空模式**：将文件状态序列化后清空缓存，随后在严格预算内恢复关键状态：
- 文件恢复预算：最多 5 个文件，单文件 5K tokens，总预算 50K tokens。
- 技能恢复预算：单技能 5K tokens，总预算 25K tokens。
*工程洞察：不全部恢复是为了避免“压缩刚节省的空间立刻被填满”的荒谬循环。优先恢复最近修改的文件和核心配置。*

### 2.5 三种微压缩机制
不调用 LLM 的轻量级上下文修剪，用于延缓全量压缩的到来：
| 机制 | 触发方式 | 执行位置 | 缓存影响 | 策略细节 |
| :--- | :--- | :--- | :--- | :--- |
| 基于时间的微压缩 | 距上次消息间隔超阈值 | 客户端 | 破坏缓存前缀 | 丢弃过期的工具结果，保留最新状态 |
| 缓存微压缩 | 可压缩工具数超阈值 | 服务端 (`cache_edits`) | 保持缓存前缀 | 将历史工具结果替换为 `[Result omitted]` |
| API Context Mgmt | API 侧 `input_tokens` 超阈值 | 服务端 | API 层管理 | 由底层 API 自动修剪不可见内容 |

*核心洞察*：缓存已过期时，先清掉旧内容再重写，让重写的内容更小更便宜。

### 2.6 Token 预算三级入口闸门
在内容进入上下文窗口前，通过三级闸门严格控制其大小：
1. **单工具结果**：超 50K 字符时持久化到磁盘，模型仅收文件路径 + 前 2KB 预览。
2. **单消息总量**：所有工具结果聚合不超过 200K 字符。
3. **Token 计数追踪**：通过规范 API 或粗略估算（字符数/4）追踪使用量。

**持久化阈值优先级链**：
`maxResultSizeChars: Infinity` (永不持久化，如 Read 工具) > GrowthBook 覆盖值 > 工具声明值 (取 `min(声明值, 50K)`) > 默认 50K。

## 三、深度防御安全与权限体系

### 3.1 六种权限模式与状态机流转
系统通过最高层控制开关定义操作边界，权限模式不仅控制执行，还影响 UI 提示符：
- `default` (无提示符): 所有工具调用需确认。
- `acceptEdits` (`>>`): 文件编辑自动通过，Shell 需确认。
- `plan` (`⏸`): 只读不写，用于架构规划。
- `bypassPermissions` (`>>`): 跳过所有检查（信任环境）。
- `dontAsk` (`>>`): 所有 ask 转 deny（CI/CD 管线，避免阻塞）。
- `auto` (`>>`): AI 分类器自动裁决。

**模式切换循环**：外部用户可通过 `Shift+Tab` 循环切换：`default → acceptEdits → plan → bypassPermissions → default`。

### 3.2 YOLO 分类器：用 AI 审核 AI
在 `auto` 模式下，系统发起独立的侧查询 API 调用，由另一个轻量级 AI 模型裁决操作安全性。
- **架构隔离**：分类器是独立的 API 调用，与主循环完全隔离，拥有自己的系统提示词、消息序列和 Token 预算。
- **上下文感知**：拥有完整对话上下文，能区分“在测试目录执行 `rm -rf`（正常构建清理）”与“在用户主目录执行 `rm -rf`（灾难性操作）”。
- **安全白名单短路**：只读操作（如文件读取、LSP 查询、Glob 搜索）零成本通过，不触发 API 调用，极大降低延迟和成本。

### 3.3 双平台内核级沙箱机制
作为安全体系的最后硬边界，利用 OS 内核隔离机制，防止模型被 Prompt Injection 后执行恶意代码：
- **macOS (Seatbelt / `sandbox-exec`)**: 通过 Profile 规则控制路径访问，支持通配符匹配（如允许 `/Users/*/repos/*`），结合 Unix socket 过滤网络访问。
- **Linux/WSL2 (Bubblewrap + seccomp)**: 只读根挂载 + 可写白名单 `bind-mount`，通过 seccomp 过滤危险系统调用（如 `ptrace`, `mount`）。

*关键差异*：macOS 的 Seatbelt 更灵活但配置复杂，Linux 的 Bubblewrap 更安全但只能精确 `bind-mount` 指定目录。

### 3.4 企业级托管安全扩展（补充）
针对远程托管场景（如 K8s 集群部署），架构支持凭证隔离机制：Agent 进程仅能看到占位凭据（如 `{{API_KEY}}`），真实 API 密钥在网络边界由 Vault Proxy 做线级替换，彻底杜绝 Agent 进程被攻破后泄露企业密钥的风险。

### 3.5 Hooks: 26 个生命周期拦截点
用户可在 Agent 生命周期的 26 个事件点注册自定义逻辑：
- **工具执行生命周期**：`PreToolUse` (可阻塞、修改输入参数), `PostToolUse`, `PostToolUseFailure`。
- **权限生命周期**：`PermissionRequest`, `PermissionDenied`。
- **会话生命周期**：`SessionStart`, `SessionEnd`。

**关键能力**：Hook 不仅能决定“是否允许”，还可**修改输入参数**。例如，注册一个 `PreToolUse` Hook，在所有 `git push` 命令前自动注入 `--no-verify`，或者在 `npm install` 前自动添加 `--registry` 指向企业内部源。

### 3.6 CLAUDE.md 四级覆盖层与热重载
用户指令覆盖模型默认行为，遵循“最后加载优先级最高”原则（利用 LLM 对最近文本的 recency bias）：
1. `/etc/claude-code/CLAUDE.md` (企业策略，全局不可变)
2. `~/.claude/CLAUDE.md` + `rules/*.md` (用户全局指令)
3. `CLAUDE.md` + `.claude/rules/*.md` (项目团队共享，提交至 Git)
4. `CLAUDE.local.md` (本地仅生效，强制加入 `.gitignore`)

系统通过 `fs.watch` 监听 `CLAUDE.md` 变化，一旦保存，立即触发热重载，重新构建系统提示词并使相关缓存失效，无需重启会话。

## 四、智能体编排与多协同模式拓扑

### 4.1 三种 Agent 派生模式与上下文隔离
从轻量到重量，系统支持三种派生模式：
1. **子 Agent**: 继承主 Agent 上下文，同步执行并返回结果。适合独立的、需要深度搜索的子任务。
2. **Fork 模式**: 独立上下文，自动后台化。适合并行探索多个方案。
3. **协调者模式**: 独立上下文，可指定模型，拥有独立生命周期。适合长期运行的后台服务。

所有派生通过 `AgentTool` (别名 `Task`) 完成。**输入 Schema 动态组合**：模型看到的参数列表精确反映当前可用能力，避免无效选项污染决策。

**AsyncLocalStorage 上下文隔离**：同一 Node.js 进程内多个并发 Agent 的身份信息、权限状态和遥测上下文通过 `AsyncLocalStorage` 隔离，防止相互污染。

### 4.2 Teams 平面协作架构
与层级派生不同，Teams 创建扁平结构的团队，Agent 通过消息传递协作。
- **架构约束**：队友不能派生其他队友（团队名册是扁平数组），防止进程树无限膨胀。进程内队友不能派生后台 Agent（生命周期绑定在 Leader 进程上）。
- **消息路由协议**：`to` 字段支持队友名称（单播）、`*`（广播）、`uds:`（本地对等通信）、`bridge:`（远程控制通信）。

### 4.3 Ultraplan: 远程规划卸载与状态机
将计划阶段卸载到 CCR (Claude Code Remote) 远程容器执行，强制使用 Opus 级模型。本地终端保持可用，用户可继续其他操作。

**架构执行流**：关键字检测 → 彩虹高亮 UI 提示 → `/ultraplan` 命令解析 → 远程会话创建 → 后台轮询（每 3 秒）→ 状态机检测。
**状态机流转**：`running` (规划中) → `needs_input` (需要用户补充信息) → `ready` (规划完成，待执行)。

### 4.4 推理深度控制三剑客
三个独立协作的机制，控制模型的“思考”深度：
- **Effort**: `low`, `medium`, `high`, `max` 四级（最高级仅特定 Pro 模型支持），直接影响 API 请求中的 `thinking_budget`。
- **Fast Mode**: 降低延迟的快速响应模式，可能牺牲部分质量。
- **Thinking**: 显式触发的深度思考链控制。

**优先级链**：环境变量 `CLAUDE_CODE_EFFORT_LEVEL`（最高，用于 CI/CD） > `AppState.effortValue`（`/effort` 命令或 UI 切换） > 模型默认值。
*关键设计*：请求构建时锁存这些动态值，确保缓存键稳定性，同时防止用户在模型思考过程中切换模式导致行为不一致。

## 五、跨会话记忆与知识沉淀架构

### 5.1 六层跨会话记忆系统
从遗忘到持久学习的完整闭环，彻底解决“金鱼记忆”问题：
| 子系统 | 核心文件 | 频率 | 职责 |
| :--- | :--- | :--- | :--- |
| **Memdir** | `memdir/memdir.ts` | 每次会话加载 | `MEMORY.md` 索引 + 主题文件，注入系统提示词 |
| **Extract Memories** | `services/extractMemories/` | 每轮结束 | Fork agent 自动提取记忆，识别用户偏好与项目规律 |
| **Session Memory** | `services/SessionMemory/` | 定期触发 | 滚动会话摘要，用于上下文压缩 |
| **Persistence** | `utils/sessionStorage.ts` | 每消息 | JSONL 会话记录存储与恢复，支持崩溃恢复 |
| **Agent Memory** | `tools/AgentTool/agentMemory.ts` | Agent 生命周期 | 子 Agent 持久化 + VCS 快照，记录探索路径 |
| **Auto-Dream** | `services/autoDream/` | 每日/空闲 | 夜间记忆整合与修剪，消除冗余，强化重要模式 |

### 5.2 记忆即文件
跨会话记忆以 Markdown 文件形式存储，而非黑盒向量数据库。
**Memdir 路径优先级**：
1. `CLAUDE_COWORK_MEMORY_PATH_OVERRIDE` 环境变量
2. `autoMemoryDirectory` 设置（仅限受信任来源，防止路径注入）
3. 默认路径：`~/.claude/projects/<项目>/memory/`

*设计决策*：同一仓库的所有 worktree 共享同一个记忆目录——记忆是关于项目的，不是关于工作目录的。用户可以直接用 `vim` 编辑 `MEMORY.md`，修改会在下一次会话生效，实现人机共编记忆。

### 5.3 Auto-Dream 机制深度解析
夜间或用户长时间空闲时，系统自动触发“做梦”机制：
1. 扫描近期的会话记忆和 Extract Memories。
2. 识别重复的模式（如“用户总是要求用特定的测试框架”）。
3. 将这些模式抽象为更高层级的规则，写入 `MEMORY.md`。
4. 修剪过时或低频使用的细枝末节记忆，控制记忆库体积。

## 六、可扩展生态：MCP 与 Skills 双轨制

### 6.1 技能系统：可调用的提示词模板
技能本质是结构化的 Markdown 知识，不是可执行代码。
**技能定义关键字段**：
- `name`: 调用名，对应 `/name` 语法。
- `whenToUse`: 告诉模型何时主动调用（出现在 `system-reminder`）。
- `allowedTools`: 执行期间自动授权的工具列表，避免权限弹窗打断流程。
- `context`: `inline`（注入主对话流）或 `fork`（子 Agent 运行，隔离上下文）。
- `files`: 附带的参考文件，首次调用时提取到磁盘。
- `getPromptForCommand`: 核心：生成注入上下文的提示词内容。

技能数量通过 1% context window 预算进行限制，防止提示词爆炸。

### 6.2 插件系统：扩展架构的顶层容器
插件清单验证 Schema 占 1681 行，由 11 个子 Schema 组合而成，包含 Metadata, Hooks, Commands, Agents, Skills, OutputStyles, Channels, MCP Server, LSP Server, Settings, UserConfig。
**安全设计**：
- 所有文件路径必须以 `./` 开头，禁止 `..`（防路径遍历攻击）。
- 市场名称保留机制，禁止冒充官方名称（如 `claude-code-official-*`）。
- 版本化缓存隔离不同插件版本，防止升级导致缓存污染。

### 6.3 MCP (Model Context Protocol) 外接大脑
MCP 如同“AI 应用程序的 USB-C 端口”，提供了将 AI 模型连接到不同数据源和工具的标准化方式。通过 MCP Server，Claude Code 可安全接入外部数据库、API 或自定义工具集，实现能力的无限横向扩展，且遵循统一的权限与遥测规范。

### 6.4 89 个 Feature Flag 门控的未发布功能
Feature Flag 通过 Bun 构建时的 `feature()` 函数实现，配合 AST 级别的 dead code elimination 将未启用分支完整移除，实现零运行时开销。
**引用计数与成熟度推断**：
- **100+ (深度集成)**: `KAIROS` (154次, 可能是时间序列/事件回溯系统), `TRANSCRIPT_CLASSIFIER` (107次, 对话意图分类器)。
- **30-99 (功能完整)**: `TEAMMEM` (51次, Teams协作), `VOICE_MODE` (46次, 语音交互)。
- **1-2 (原型阶段)**: `ULTRATHINK` (1次, 超深思考模式)。

五大功能域：自主 Agent 与后台运行（18 个）、远程控制与分布式执行（14 个）、上下文管理与性能优化（17 个）、多模态交互（15 个）、企业安全与审计（25 个）。

## 七、架构设计核心原则总结与普适性启示

1. **极薄内核与厚外壳**：模型负责概率推理，Harness 负责确定性兜底。不要在 Agent 中写复杂的状态机，让模型来决定状态转移。
2. **分段与缓存优先**：将大型提示词拆分为可独立缓存的小段，通过“行李箱打包”最大化前缀匹配，通过 `cacheBreak` 标记区分稳定与易变内容。
3. **分层纵深防御**：应用层权限（软边界，灵活可配） → AI 决策层审核（YOLO 分类器，理解意图） → OS 内核层沙箱（硬边界，物理隔离）。
4. **可观测性内置**：每个关键子系统配备遥测事件、25 种错误分类和缓存中断检测。没有可观测性的 Agent 是黑盒，无法在生产环境运行。
5. **锁存防止缓存中断**：请求构建时锁存动态值（如 Fast Mode、AFK 状态），确保缓存键稳定，不惜牺牲一帧的实时性换取 90% 的成本降低。
6. **快照-清空-恢复模式**：压缩前保存现场、清空缓存、有预算地恢复最关键状态。避免“压缩节省的空间立刻被填满”的循环。
7. **动态 Schema 组合**：模型看到的参数列表精确反映当前可用能力，避免无效选项污染决策，提高工具调用准确率。
8. **构建时特性门控**：通过 `feature()` + dead code elimination 实现零运行时开销的功能开关，敢于在代码库中保留半成品功能。
9. **记忆即文件**：跨会话记忆以人类可读的 Markdown 存储，拒绝黑盒向量数据库。实现人机共编记忆，信任用户对系统的理解与控制权。
10. **设计摩擦作为安全阀**：通过 API 命名（如 `DANGEROUS_` 前缀）和强制参数（如 `reason`）制造开发摩擦，防止工程师无意识地破坏关键约束（如缓存前缀）。
